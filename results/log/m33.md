# M33 — the shared-readout / channel-independence decomposition on the SYNTHETIC mixer-win tasks: on the clean single-flag axis the attribution FLIPS vs forecasting — here token-MIXING is ~the entire win (the CI/readout legs are bundled/untestable here, see caveats)

M31/M32 decomposed the **forecasting** `trm_mixer` win into four ingredients — cross-cell
token-**MIXING**, **channel-INDEPENDENCE**, a per-cell **shared READOUT**, per-variable
**WEIGHT-SHARING** — and found the win was NOT mixing (net harmful on weather); it was the
channel-independent parameterization (CI-led, shared readout a modest second). CLAUDE.md §11.2
#8/#13/#14 then *asserted*, without measuring, that the SYNTHETIC constraint-coupled mixing wins
(Sudoku M23, `converge` M24, `hopfield` M24c, `disruption` M24f, `mixed_converge` M28a) are
"untouched" and remain genuine **mixing** wins. **M33 measures that** — it runs the exact M32 7-arm
decomposition on all five synthetic mixer-win tasks.

**No code changes.** All four control arms already exist as off-by-default `TRMMixer` flags
(`trm_mixer_nomix`, `trm_mixer_unsharedro`, `trm_mixer_nomix_unsharedro`, `trm_mixer_nomix_distinctw`;
`src/looptab/models/mixer.py`), byte-identical to `trm_mixer` at defaults. Pure config + budget-probe
+ run. Control-arm widths solved to `trm_flat`'s budget by `scratchpad/m33/probe_budget.py`; the
geometry-varying tasks are split one config per width (converge/hopfield/mixed_converge/disruption)
because the `trm_flat` budget shifts with the output width; Sudoku stays a single `n_givens` sweep
(its encoding is constant: in=252, cells=36, num_classes=6). 10 configs, 8 seeds each (Sudoku 6),
`num_threads=1`.

**Sign convention (OPPOSITE of M31/M32).** These are classification tasks, so **POSITIVE Δ favours
the first arm** (accuracy / exact-match) — the reverse of the negative-Δ-favours-first MSE in
forecasting. The within-run additive identity is
`Δ(mixer − ff) = Δ(mixer − nomix)[MIXING] + Δ(nomix − ff)[CI + readout + weight-share]`.

## Headline

On every one of the five constraint-coupled task families, the decomposition is a **clean mirror
image of forecasting**:

- **Token-MIXING is essentially the ENTIRE win (the ONE clean single-flag leg)** —
  `Δ(trm_mixer − trm_mixer_nomix)` EM **+0.51 … +0.99, 8/0 (Sudoku 6/0), p≤.031** in every cell. This is
  the only comparison here that flips exactly one flag (`disable_token_mix`) with everything else held,
  so it is the load-bearing result: removing cross-cell mixing collapses the win. The mechanism CLAUDE.md
  #7/#8 credited — now measured, not asserted.
- **A channel-independent looped parameterization UNDERPERFORMS a joint MLP** —
  `Δ(trm_mixer_nomix − ff_matched)` EM **negative in every cell, 0/8** (−0.02 … −0.64). Read this as a
  DIRECTION, not a clean isolation: `nomix` differs from `ff` in more than channel-independence (looped
  vs feedforward, per-cell tokenized vs flat, shared per-cell channel MLP vs one joint MLP) — the same
  bundle M31/M32 flagged on the forecasting side (`Δ(nomix − ff)` = CI + readout + weight-share + loop).
  So the honest claim is *the channel-independent looped arm that WON forecasting is worse than a plain
  feedforward on these coupled targets* — consistent with "cutting the cross-cell path hurts where a cell
  needs its neighbours," but bundled, not a single-flag CI measurement.
- **Shared READOUT and WEIGHT-SHARING are UNTESTABLE here (degenerate saturation), not measured-inert** —
  `Δ(nomix − nomix_unsharedro)` = `Δ(nomix − distinctw)` = 0.000 EM, but this is because the three
  channel-independent arms **converge to the identical per-cell decision**: their per-seed metrics are
  bit-identical to 15 sig figs (even `coherence_excess = 8.7827e-06` matches across all three arms). On
  these tasks the CI optimum is a trivial per-cell lookup (`converge` has `cell_dim = in/out = 1` — each
  cell sees one bit + its position), and every readout/weight parameterization represents that lookup with
  the **same argmax**, so the deltas have **zero power** to detect a readout effect. The repo's own
  `sign_test_robust` flags exactly this: every RO/WS leg is `n_near_tie = 8, sign_p_robust = 1.0`
  (all-ties). So this is *consistent with* the readout/weight-share being forecasting-specific (there the
  M×H readout balloons with horizon and the parameterization matters), but it is **not evidence** for it —
  the axes simply cannot be measured where the CI arm degenerates.

The additive identity `Δ(mixer−ff) = Δ(mixer−nomix) + Δ(nomix−ff)` holds to machine epsilon (residual
≈−1e-16) in all 12 cells — but that is an arithmetic tautology (linearity of the mean over paired per-seed
deltas), a consistency check on the bookkeeping, **not** independent evidence. What it makes explicit: the
headline mixer win decomposes as mixing (over-explaining it) plus a negative CI-bundle term (dragging it
back).

## Reproduction guard (the fork did not perturb the baseline arms)

`run.py`'s C3 reseeds before each arm, so adding the four control arms leaves `trm_mixer` / `trm_flat`
/ `ff_matched` on identical data + init. Confirmed: every `HEAD = Δ(mixer − ff)` reproduces its shipped
milestone (converge/hopfield/mixed_converge/Sudoku EM match to ≤0.002; disruption w24 accuracy +0.041
matches the M24f headline +0.041 exactly, and w32 reproduces once corrected to `gamma=15` — see the note
above; the M24f w32 headline is accuracy +0.034). `Δ(trm_flat − ff)` also reproduces (flat ≈ ff on
Sudoku/disruption; flat > ff on converge — the leg-2 edge M24 flagged). Note this is a per-cell
*reproduction of the delta*, not a per-seed bit-identical rerun of the baseline — the baselines used a
3-arm config, and the sign/magnitude of each shared arm's Δ matches within rounding.

## Reading

1. **#8's dividing line — the MIXING axis — is now MEASURED, not asserted.** On the clean single-flag
   comparison, the mixer's contribution *flips sign* between regimes: forecasting (M31/M32) mixing is net
   harmful; synthetic constraint-coupled (M33) mixing is ~the entire win (+0.51…+0.99 EM, 8/0). That flip
   is the load-bearing, cleanly-isolated result. The accompanying reads are weaker: the CI-arm-underperforms
   direction is bundled (point 2), and the readout/weight-share axes are untestable here (point 3). **Same
   mixing operator, opposite sign of contribution**, set by whether the outputs are genuinely cross-cell
   coupled — a strong confirmation of #7/#8 and the sharpest statement of the forecasting exception #13/#14.
2. **Why the CI arm underperforms here (mechanistic, but bundled).** `trm_mixer_nomix` is a
   channel-independent looped MLP: cell *i* only ever sees its own input column. On a ring/grid/clique
   fixed point, a cell's correct output *depends on its neighbours*, so cutting the cross-cell path forces
   each cell to guess from local input alone — worse than the joint `ff` that at least sees the whole row.
   On forecasting the opposite holds: each variable is best predicted from its own history, so cutting
   cross-variable leakage *helps* (the DLinear/PatchTST channel-independence intuition). Caveat as above:
   `Δ(nomix − ff)` bundles channel-independence with the loop, per-cell tokenization, and the shared
   channel MLP, so this is the direction and a plausible mechanism, not a single-flag isolation of CI.
3. **The readout/weight-share axes are untestable on these tasks (not a measured null).** All three
   channel-independent arms converge to the *identical per-cell decision* (per-seed metrics bit-identical
   to 15 sig figs), because the CI optimum is a trivial per-cell lookup and every parameterization gives
   the same argmax. So `Δ(nomix − nomix_unsharedro)` = 0.000 in **EM** carries no information — and on
   Sudoku (num_classes=6, the one place a readout effect *could* show) the **accuracy** delta is in fact
   nonzero but negligible (+0.0004 / −0.0003, noise around 0); only EM floors to 0. This is *consistent
   with* the M31/M32 finding that the shared readout's value is forecasting-specific parameter-efficiency
   (the M×H readout ballooning with horizon), but M33 does **not** independently measure it.
4. **Width nuance on converge (does not change the story).** As w grows the mixing leg stays ~+0.97 EM
   (constant) while the CI penalty shrinks (−0.64 → −0.30 → −0.02) and the headline grows (+0.345 →
   +0.944). That is because `ff` itself collapses on the wider ring, so `nomix` (also channel-independent)
   is less far behind it — but mixing carries essentially the whole win at every width regardless.

## Caveats / bounds

- **`trm_mixer_nomix_distinctw` budget breach (documented, moot).** The per-cell distinct-weight arm's
  param count jumps coarsely, so it lands ~0.905–0.936 of `trm_flat` (CONSERVATIVE / under-budget) on the
  w24/w32 binary cells — but `Δ(nomix − distinctw) = 0.000` there anyway, so the breach cannot affect the
  conclusion. `trm_mixer` at converge_w48 sits at 0.866, the inherited baseline conservative breach
  (unchanged from the shipped M24 config). All matched arms are within the ±5% spec; the mixer arms
  inherit the baseline mixer widths so `trm_mixer` runs up to 1.041 (converge_w24) — within spec, not
  ±1.5%. The control arms (`nomix`/`unsharedro`/`nomix_unsharedro`) are within ±1% by construction.
- **Scope.** Five task families, tiny CPU models, the same lean recipe as the baselines. The firm claim is
  the sign-flip of the **mixing** contribution (clean single-flag `Δ(mixer − nomix)`) between
  coupled-synthetic and channel-independent-forecasting regimes, established on 12 cells with 8/0 (6/0)
  sign tests — not a scaling claim, and not a clean isolation of the CI or readout axes (see below).
- **RO/WS legs are degenerate here (NOT a code bug, but NOT a measured null either).** The three
  channel-independent arms (`nomix`, `nomix_unsharedro`, `nomix_distinctw`) are genuinely distinct models
  (different param counts, different logits at init — confirmed dynamically), but they converge to the
  *identical per-cell decision*, so their per-seed metrics are bit-identical to 15 sig figs. That means the
  readout and weight-share deltas have zero measurement power on these tasks (the CI optimum is a trivial
  per-cell lookup with the same argmax under any parameterization). `sign_test_robust` correctly reports
  these as all-ties (`n_near_tie = 8`, `sign_p_robust = 1.0`). So "readout/weight-share = 0.000" is a
  saturation artifact, consistent with but not evidence for their forecasting-specific value. The
  byte-identity flag tests in `tests/test_models.py` cover the *defaults*, not this convergence.
