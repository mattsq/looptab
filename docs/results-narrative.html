looptab · results narrative

Does "thinking longer" help a model that just fills in a table?

A research program tested whether iterative latent refinement — the loop at the heart of recent reasoning models like TRM and HRM — gives a small tabular model an edge over a parameter-matched feedforward network with no loop at all. Thirty milestones of synthetic and real experiments later, the answer is a specific, evidence-backed it depends — not a vague one. This page walks through the seven findings that hold up.

29 milestones, ~2,000+ trained model runs every number below ships with its control and a paired sign test synthetic suite + 4 real datasets

Chapter 1

The question, and the rule that makes the answer trustworthy

Models like TRM and HRM don't just do one forward pass — they maintain a hidden "scratchpad" state and refine it in a loop: update the scratchpad, re-read an answer from it, repeat, for a fixed number of steps. The idea is that more refinement steps let the model do something closer to reasoning than a single-shot guess.

That loop is also exactly the kind of thing that's easy to fool yourself about. A loop with N steps and a hidden state of size h has, roughly, the same capacity as a plain feedforward network that is N layers deep — so any accuracy gain the loop shows could just be "it's secretly a deeper, wider network," not "the loop is doing something a non-recurrent network structurally can't."

So every experiment in this program follows one rule: a recurrent result never ships alone. It always ships next to a parameter-matched feedforward network (same weight budget, one forward pass, no loop) and, where relevant, a depth-matched stack of the same block repeated without sharing weights. The number that matters is never the loop's raw accuracy — it's the gap (Δ) between the loop and its matched control, with enough seeds (≥5, usually 8–10) to run a paired sign test on that gap.

Why 8–10 seeds, not 5? Five seeds is a variance floor, not a significance floor: the exact two-sided binomial sign test can't reach p<.05 with fewer than 6 seeds (a unanimous 5/5 split still only gets p=.0625). Every "beats its control" claim on this page needing a real significance call uses 8 or 10 seeds specifically so a unanimous result can clear that bar — the same spirit as the NLP-reporting literature's warning that a conclusion can flip depending on how much search/compute went into reaching it.

The testbed is a small synthetic suite with a difficulty knob and a known ground-truth algorithm — parity (an irregular target buried in noise bits), an elementary cellular automaton run for T steps (iterated, a target that is an explicit multi-step algorithm), and CA variants that are run to a fixed point (converge) — plus, later, real tabular datasets. Two real bridges close out the story: real multi-label classification and real multivariate forecasting.

Chapter 2

A loop built the obvious way is close to a deep net in disguise

The simplest recurrent design updates one flat MLP over the entire row at once — every step sees the whole table row as a single flattened vector. That design does edge out a matched feedforward net on fixed-point targets (a real effect — see Chapter 3): it's measurably more accurate cell-by-cell, and that per-cell edge holds steady to growing as the row widens. On whole-row exact match specifically the edge looks like it shrinks — that's an artifact of how exact match compounds per-cell accuracy across more cells, not a sign the underlying edge is disappearing. Either way, it's dwarfed by what actually turns out to matter (Chapter 4).

Chapter 3

The mechanism that is real: refining every cell in one shared state buys whole-row coherence

Isolate one design choice: does the loop refine all output cells inside one shared latent, or does each cell get its own independent latent that never talks to the others? On local fixed-point targets, the shared-latent version wins decisively on whole-row exact match — and, unusually for this program, the edge grows with model size instead of shrinking or staying flat.

Chapter 4

The pivot: what actually solves hard puzzles is letting cells talk to each other — not the loop

Every result up to here used the flat, whole-row update. Swap it for an update that lets each table cell attend to every other cell — a cross-cell mixing operator — and the picture changes completely. On Sudoku, the canonical hard-reasoning benchmark for this whole model family, the flat loop and the matched feedforward net both fail outright on hard puzzles. The mixing loop solves the majority of them.

0%→89.3% Hard Sudoku (14 givens), whole-puzzle exact match: feedforward and flat loop both fail outright; the cross-cell mixer solves it.

The obvious next question: is that a win for recurrence, or just for the mixing architecture? Build the honest control — a mixing block stacked N times without sharing weights, at the same parameter budget — and run it against the weight-tied loop on the fixed-point CA task from Chapter 2 (the cleanest task where that control exists).

The untied stack, at matched budget, already captures almost the entire win over feedforward. The weight-tied loop keeps a real, budget-clean edge over it — but that edge is the ordinary parameter-efficiency benefit of weight sharing, not a new capability: give the untied stack ~7× the parameters and it fully closes the gap.

Chapter 5

Where mixing helps: table cells that are genuinely coupled to each other

Cross-cell mixing isn't a free lunch. It helps exactly when the output cells depend on each other — whether that coupling is local (a ring of neighbors) or dense (every cell coupled to every other cell, as in a Hopfield-style attractor network) — and does nothing or actively hurts when the outputs are exchangeable, independent quantities with no structure to propagate.

Chapter 6

Two real-data bridges, and they land on opposite sides of that line

Synthetic findings are only useful if they survive contact with real tabular data. Two real bridges were run, chosen specifically to fall on either side of the Chapter 5 divide.

Real multi-label classification

micro-F1, mixer vs. matched feedforward · 10 seeds, both ties

Real multivariate forecasting

MSE reduction, mixer vs. matched feedforward · 10 backtest blocks, both p<.03*

Negative on multi-label: a naive feature-to-label reshape has no shared input/output structure for cells to propagate across — labels are just a flat output vector, not cells on a shared grid. The mixer ties a plain joint MLP on micro-F1 and accuracy on two datasets; every mixer-specific edge that looked real was an exact-match artifact of the label-combination distribution, and disappeared under F1.

Positive on forecasting: multivariate time series supply exactly the structure multi-label lacked — variable i on the input is the same variable i on the output, a genuine shared cell topology. There the cross-variable mixer beats a matched MLP on MSE, and the edge grows with the number of coupled variables — the clearest real-data positive in the whole program.

Rigor. Multi-label: 10-fold CV, micro+macro F1, 2 datasets (yeast, scene), both ties. Forecasting: 2 datasets (ETTh1, Weather), MSE/MAE/R², both significant. *The forecasting sign test is indicative, not clean: the 10 backtest blocks have disjoint test windows but nested/overlapping training windows, so the per-block deltas are correlated rather than independent — the exact binomial p-value is anti-conservative. The effect is consistent across easy and hard blocks (real), but treat the p-value as suggestive.

Chapter 7

The honest ceiling: it never finds a real fixed point, and a hierarchy wasn't earned

Two findings bound how far to trust the loop, even in its winning regime. First, introspection: even where the loop wins, its latent state doesn't actually settle into a fixed point across refinement steps — the residual between steps stays large, and unrolling past training length makes it worse, not better. It behaves like a deep network computing in stages, not like an algorithm converging to an answer.

Iterated-CA task — accuracy vs. test-time loop count, trained at 4 steps

The same shape of chart that motivated this whole research program — run on our own task, it goes the other way

Rigor. The ARC-Prize HRM autopsy — the paper that set this whole program's "test the loop before the hierarchy" approach — found that giving HRM one extra refinement loop nearly doubled its accuracy (18.6%→35.5%), with further loops adding diminishing-but-positive returns (8 loops: 38.1%; 16 loops: 39.0%). Asking the identical question here — same model, same task, only the number of test-time loops changes — gives the opposite shape: accuracy peaks exactly at the trained loop count (97.2%) and collapses toward chance the moment test-time loops exceed it, then never recovers. (Deeper training depths don't even reach a peak to collapse from — they hit a separate learnability wall first, unrelated to this effect.) Canonical: m1_iterated_extrapolation_20260620T023255.

Second, the hierarchy question — the "high-level slow / low-level fast" split that HRM introduced. The instinct to build a two-timescale version of this loop was tested against its own falsification criterion first, the way the original ARC-Prize post-mortem on HRM recommended: only add the second timescale if the single loop provably plateaus below the target at equal training compute, in a way a feedforward control doesn't share. It did plateau — but adding more training data, not a second timescale, closed the gap:

Two-timescale fixed-point task — exact match vs. training-set size

Weight-tied loop vs. matched feedforward, same architecture, only training data grows

At 64k training rows the single-timescale loop solves the task outright (99.0% exact match). The apparent need for a hierarchy was a sample-size wall shared by the feedforward control, not a timescale-specific deficit only a hierarchy could fix — so the hierarchical architecture was not built. Building it anyway would have repeated the exact mistake the HRM autopsy diagnosed: adding structure before proving the simpler model can't already do the job.

In one page

What this loop buys on tabular data, and what it doesn't

It buys

  • Whole-row coherence from refining all outputs in one shared latent state, growing with model size, on local fixed-point targets.
  • Hard constraint-satisfaction solving (Sudoku, ring/fixed-point CA) once the update lets cells mix — a real, large, growing-with-difficulty win over a matched MLP.
  • A generalization win on real multivariate forecasting — the mixer fits training data worse but tests better, the signature of a genuinely better inductive bias, not overfitting.
  • Parameter efficiency from weight-tying — a real, if modest, edge over an untied stack at matched budget.

It doesn't buy

  • A capability an untied version of the same mixing architecture lacks, once capacity is matched — the win is the architecture, not the recurrence.
  • A literal converging fixed point — the trained loop's latent state doesn't contract, even where it wins outright.
  • Depth-extrapolation — more test-time refinement steps do not recover longer computations than it was trained on.
  • Any edge on exchangeable features or real multi-label classification — tasks with no shared input/output cell structure for cells to propagate across.
  • Justification for a hierarchical (two-timescale) extension — the one instance tested was a data wall, not a case for a hierarchy.
